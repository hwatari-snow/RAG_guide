Cortex Search コスト詳細

1. EMBED_TEXT トークンコンピュート（埋め込み処理）

Cortex Search は、検索カラム内の各テキスト行を自動的にベクトル空間に埋め込みます。行の挿入・更新のたびに埋め込みコストが発生しますが、増分処理されるため、変更分のみが課金対象です。

コスト = (Embedding モデルのクレジットレート / 1M トークン) × 行数 × 行あたりトークン数

計算例: 公式ドキュメントの大規模ケース

モデルレート: 0.05 credits / 1M tokens 行数: 10,000,000 行 × 500 tokens/行 合計: = 5,000,000,000 tokens ÷ 1,000,000 = 250 credits

計算例: 本デモ（景品表示法ガイドライン 33 PDF）

チャンク数: 約 1,000 チャンク チャンクサイズ: 1,500 文字 ≈ 375〜500 tokens（日本語） 合計トークン: ≈ 500,000 tokens Embeddingコスト: 0.05 × 0.5 ≈ 0.025 credits
ポイント: 小規模データ（1,000チャンク程度）では、Embeddingコストはほぼ無視できるレベルです。

2. Serving コンピュート（サービング＝常時課金）

Cortex Search のサービングは、インデックスされたデータの GB/月 単位で課金されます。サービスが起動中（resumed）である限り、クエリの有無に関わらず課金されます。

月額コスト = (クレジットレート / GB) × 行数 × (ベクトル次元 × 4 bytes + 行データサイズ) ÷ 1,000,000,000

計算例: 公式ドキュメントの大規模ケース

パラメータ: 10M行、768次元、1,000 bytes/行、6.3 credits/GB 計算: 6.3 × 10M × (768 × 4 + 1,000) / 1B = 256.5 credits/月

計算例: 本デモ（~1,000 チャンク）

パラメータ: 1,000行、768次元、~2,000 bytes/行、6.3 credits/GB インデックスサイズ: 1,000 × (768 × 4 + 2,000) = 5,072,000 bytes ≈ 0.005 GB 月額: 6.3 × 0.005 ≈ 0.03 credits/月
ポイント: 本デモ規模では Serving コストも極めて小さいです。ただしサービスが常時起動している限り課金されるため、開発中は ALTER CORTEX SEARCH SERVICE ... SUSPEND で停止しましょう。

3. Warehouse コンピュート（リフレッシュ）

Cortex Search のインデックスリフレッシュに Virtual Warehouse が使用されます。TARGET_LAG の設定に応じてリフレッシュ頻度が変わります。

  • MEDIUM ウェアハウスで十分（大半のケースでLARGE以上は不要）
  • データ変更がなければ Warehouse クレジットは消費されない
  • TARGET_LAG を適切に設定してリフレッシュ頻度を最適化

4. ストレージ

ソースクエリのマテリアライズ結果と検索インデックスの保存に使用。TB単位の定額制。小規模データでは無視できるレベル。

Cortex Agent コスト詳細

課金モデル

Cortex Agent は使用トークン数に基づく従量課金です。以下の要素で構成されます：

コスト要素 説明 課金基準
オーケストレーション Agent の計画立案・タスク分割・ツール選択・リフレクションに使用される LLM 推論 入力 + 出力トークン数
Cortex Search 呼び出し 非構造化データの検索ツール実行。検索クエリのEmbedding + 結果取得 Search サービングコスト + クエリEmbedding
Cortex Analyst 呼び出し
（使用時のみ）		 構造化データのSQL生成。セマンティックモデルに基づく text-to-SQL トークン数
Warehouse 実行 Agent のツール実行環境として Warehouse を使用 WH サイズ × 稼働時間

Agent のトークン構成

1回の Agent リクエストで消費されるトークンの内訳：

トークン種別 内容 概算規模
入力トークン システム指示 + Agent設定 + ユーザー質問 + 検索結果コンテキスト + スレッド履歴 数千〜数万 tokens
キャッシュ読取トークン 同一スレッドの前回コンテキストのキャッシュ利用分（低コスト） 可変
出力トークン Agent の回答テキスト + 参考資料リンク 数百〜数千 tokens

利用可能モデルとコスト

Agent に使用するモデルによりトークンあたりのクレジットレートが異なります。具体的なレートは Snowflake Service Consumption Table を参照してください。

前提条件

コスト監視クエリ

Cortex Search の利用状況

-- Cortex Search の日次利用履歴
SELECT *
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_SEARCH_DAILY_USAGE_HISTORY
WHERE START_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
ORDER BY START_TIME DESC;
-- Cortex Search の時間別サービングクレジット
SELECT *
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_SEARCH_SERVING_USAGE_HISTORY
WHERE START_TIME >= DATEADD('day', -7, CURRENT_TIMESTAMP())
ORDER BY START_TIME DESC;

Cortex Agent の利用状況

-- Cortex Agent のリクエスト別利用履歴
SELECT
  START_TIME, USER_NAME, AGENT_NAME,
  TOKEN_CREDITS, TOKENS,
  TOKENS_GRANULAR, CREDITS_GRANULAR
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AGENT_USAGE_HISTORY
WHERE START_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
ORDER BY START_TIME DESC;
-- Agent の日次トークン消費サマリー
SELECT
  DATE_TRUNC('day', START_TIME) AS usage_date,
  AGENT_NAME,
  SUM(TOKEN_CREDITS) AS total_credits,
  SUM(TOKENS) AS total_tokens,
  COUNT(*) AS request_count
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AGENT_USAGE_HISTORY
WHERE START_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY 1, 2
ORDER BY usage_date DESC;

Cortex AI Functions（AI_PARSE_DOCUMENT 等）の利用状況

-- AI Functions の日次クレジット消費
SELECT
  DATE_TRUNC('day', START_TIME) AS usage_date,
  FUNCTION_NAME, MODEL_NAME,
  SUM(CREDITS) AS total_credits,
  COUNT(DISTINCT QUERY_ID) AS query_count
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE START_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY 1, 2, 3
ORDER BY usage_date DESC;

コスト管理機能

リソースバジェット（Cortex Agent 向け）

Cortex Agent にはタグベースのリソースバジェット機能があり、月次の利用上限を設定できます。閾値に達した際にストアドプロシージャを自動実行してアクセスを制限できます。

-- 1. タグを作成してAgentに適用
CREATE TAG cost_mgmt.tags.cost_center
  ALLOWED_VALUES 'rag-demo';

ALTER AGENT DEMO.KAIBALAB.LEGAL_GUIDE_AGENT
  SET TAG cost_mgmt.tags.cost_center = 'rag-demo';

-- 2. Budgetを作成（Snowsight UIまたはSQL）
-- 月間上限を設定し、80%到達時に通知、100%到達時にアクセス停止

CORTEX_AGENT_USAGE_HISTORY ビュー

各 Agent リクエストのトークン・クレジット消費を詳細に確認できるビューです。TOKENS_GRANULAR カラムでサービスタイプ（cortex_agents / cortex_analyst）やモデル別の内訳が取得可能です。

ポイント: Account Usage ビューのデータ反映には最大60分の遅延があります。リアルタイム監視が必要な場合は INFORMATION_SCHEMA 系のビューと組み合わせましょう。
